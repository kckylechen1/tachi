# Tachi 架构重构 - 原始评审日志
**(Codex 5.4 与 Kimi 2.5 完整发言备份)**

## ==========================================
## 第一轮：关于“软合并、异步流、提取器”的底层批判
## ==========================================

### 【Codex 5.4 评审意见】
我看了 `~/Desktop/Tachi` 里这几块核心代码：`mcp/extractor.py`、`mcp/workers/*`、`mcp/event_queue.py`、`mcp/store.py`。结论如下：

**1）Tachi 方向没错 —— 成立**
它准确识别了“记忆冲突、状态冲刷、需要演化式记忆”这个问题。它试图用 `ConsolidatorWorker` 合并语义，避免状态冲刷。但它更接近“弱结构知识演化系统”，不是严格意义上的知识图谱。

**2）实现太繁琐、脆弱 —— 成立，且证据充分**
- **高度依赖 LLM 输出 JSON，天然脆**：`extractor.py` 和 `causal.py` 里直接 `json.loads(clean.strip())`，解析失败直接返回空。这在工程上意味着数据丢失是常态。
- **多层异步 worker 链条长，故障面大**：Extractor -> Consolidator -> Causal，多跳弱事务链。任何一步抖动，链条就断。没有真正的 workflow 编排保障。
- **合并逻辑是“软推断”，不是“硬规则”**：`revision` 只能保证并发不写坏数据库，但防不了“LLM 把两个状态揉成一坨半真半假的折中叙事”。
- **Reranker ROI 偏低**：在解决核心稳定性写入问题前，加 Reranker 属于锦上添花甚至转移注意力。

**3）强状态管理不该走“向量库 + LLM 软合并” —— 成立，而且是最关键的一条**
对于证券持仓、观察池等**强状态（strong state）**，它们必须是单值真相、必须可审计、必须约束更新语义（加仓/减仓，不是语义合并）。
Tachi 这种软合并只适合“软记忆”（偏好、习惯）。对于交易场景：
- **软更新（Tachi 式）**：从文本推断状态，再语义合并，追求像人类记忆。
- **硬更新（状态文件式）**：Agent 直写 `positions.json`，追求当前状态唯一、可验证。

### 【Kimi 2.5 评审意见】
**1. 方向是否正确？✅ 部分正确**
去重阈值、合并机制设计合理。但这套架构更适合"软知识"（用户偏好、项目背景），不适合强状态。

**2. 实现是否繁琐脆弱？✅ 完全正确**
- `extractor.py:83-97` 没有任何 schema 校验或 fallback 结构化解析，只靠"失败了就丢弃"。
- 异步队列只有 60 秒锁，Worker 崩溃后任务卡在 PROCESSING；无死信队列，失败无限重试。
- Reranker 额外增加 API 成本和延迟，失败时静默 fallback 导致结果不确定。

**3. 强状态管理是否应该用"硬更新"？✅ 完全正确**
- **合并非确定性**：同样两条记忆，不同时间合并结果可能不同，无法审计。
- **向量相似度不可靠**：“用户持有 AAPL 100 股” 和 “用户持有 AAPL 200 股” 语义高度相似，但状态完全不同！如果用 0.85 阈值去合并，直接导致严重 Bug。

## 第二轮：关于“MCP 层、Causal 降级”的评估
## ==========================================

### 【Codex 5.4 评审意见】
结论：**MCP 本身没问题，Causal 该保留但必须降级，真相层必须被隔离。**
- **MCP 不是病灶**：它只是接口层，负责工具暴露（`save_memory / search_memory`）。MCP 作为 Antigravity 这种工具的协议层完全成立，别误砍传输层。
- **真正的污染发生在 derived → truth 这条链**：`causal.py` 会把纠正结果直接 `save_memory(source="causal")`。推断物被提升成了近似事实层输入，这是架构错位。
- **Causal 应该保留，但降级为 Explainability 层**：它对复盘行为偏差极有价值。但必须存在单独的 `derived` 库，只能引用真相，不能反向覆盖真相。

### 【Kimi 2.5 评审意见】
- **MCP 成立**：它只是协议层，底层怎么存是实现细节。
- **Causal 降级**：它提取的是 `{context, wrong_action, correct_action}` 三元组。这是“对事实的元评论”，而不是事实本身。当前 `save_memory(importance=0.9)` 比原始事实还高，极其危险。
- **推荐改造**：增加 `layer` 字段（`source_of_truth` / `derived`），主检索只拉 truth，显式需要时才拉 derived。

## 第三轮：关于“Stock/Trading 场景是否 Fork 内核”的评估
## ==========================================

### 【Codex 5.4 评审意见】
结论：**先别 fork Tachi 内核。把 trading 做成 domain pack。**
- `watchlist` / `fund_position` 本质上是领域实体/语义，不是内核机制。如果因为这些词 fork，等于把“领域污染”误判为“内核能力差异”。
- **何时做 Domain Pack？** 当底层（存储、索引、权限、时序）一致，只是 Schema 不同时，做可插拔的 Pack。
- **何时独立成 Execution 系统？** 当系统需要承担**金钱正确性责任**（比如回答真实持仓到底多少、需要强一致账本、执行撤单风控）时，它就不再是 Memory Core 的活儿了。
- **名言**：如果它错了，只会让记忆/理解变差，那属于 domain pack；如果它错了，会让钱错、仓位错，那必须独立系统。

### 【Kimi 2.5 评审意见】
- **当前判断是对的**。Memory core 职责是存储、检索、版本、权限，这些都是 domain-agnostic 的。过早 fork 会导致后续维护成本指数爆炸。
- **Domain Pack 边界**：是「记住什么」→ memory core。是「做什么」→ 独立执行系统。
- **下一步**：把 `watchlist` 抽成 Schema + Adapter，确保 core 无 trading 硬编码。定义 domain pack 接口规范。
